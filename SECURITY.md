# Security & merge policy

This repository is public, so **anyone can open a pull request.** A brand is just
data (colors, fonts, text, image URLs), but data is still rendered into a live
web page — so an unreviewed merge is a supply-chain risk. Three layers keep the
published site safe.

## 1. Merge gate — what may be merged

A PR is only merged after confirming:

- **Author / provenance.** Brand PRs are generated by the trusted engine runner
  (bot commits touching only data). A hand-authored PR from an external
  contributor gets full code review before merge.
- **Changed paths.** A brand PR may only touch:
  `brands/`, `agents/`, `tokens/`, `briefs/`, `storybook/src/content.json`,
  `storybook/src/images.json`.
  Any change to executable code or CI — `.github/**`, `**/*.js`, `**/*.mjs`,
  `**/*.css`, `package.json`, `package-lock.json`, Storybook config — is
  **not** a brand change and requires deliberate code review. Never fast-merge it.
- **The data validator passes** (`node storybook/validate.mjs`, see layer 2).

Recommended repo settings: protect `main` (require a PR + review, disallow direct
pushes), and keep **no self-hosted runners** on this public repo — all builds run
on the private engine repo's runners, so a hostile PR can never execute on our
infrastructure.

## 2. Deploy gate — `storybook/validate.mjs`

Runs in the publish workflow **before** the Storybook build. It rejects, and
aborts the deploy, when brand data contains:

- a color that is not a literal hex value (`tokens/*.tokens.json`, `content.json`),
- an image URL that is not `https://images.unsplash.com/…` or `plus.unsplash.com`,
- injection payloads in any text field (`<script`, `javascript:`, `on…=`, …).

So even if hostile data is merged by mistake, it never reaches the live site.

## 3. Renderer hardening — `storybook/src/stories/Brand.stories.js`

Defense in depth: the renderer treats all brand data as untrusted.

- Every text value is HTML-escaped (`esc`).
- Every value injected into a `style="…"` attribute is validated: hex colors via
  `hx()`, image URLs via `safeUrl()`/`imgBg()`. A value that isn't a clean hex or
  an allowed Unsplash URL is dropped to a safe fallback — it cannot break out of
  the attribute to inject markup or event handlers.

## Reporting

Found a way around these controls? Please open a private security advisory rather
than a public issue.
